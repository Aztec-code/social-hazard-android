# Social Hazard Event Protocol

This document defines a clean JSON event protocol for the Social Hazard Android client and Spring Boot backend.

Goals:
- server-authoritative game state
- room-code multiplayer with reconnect support
- thin Android client that renders from server snapshots
- explicit, interview-friendly payloads and state transitions
- no database assumptions for v1

## 1. Transport

- Production base URL: `https://api.fluxcloud.dev`
- HTTP health check: `GET https://api.fluxcloud.dev/health`
- Production WebSocket: `wss://api.fluxcloud.dev/ws`
- Content type over WebSocket: JSON text frames
- Protocol version: `1`

## 2. Envelope Format

### Client -> Server envelope

```json
{
  "protocolVersion": 1,
  "type": "CreateRoom",
  "requestId": "req_01J9R2WY4R2Q0M4E8VJ7A3N6KD",
  "sentAt": "2026-04-15T12:00:00Z",
  "payload": {}
}
```

### Server -> Client envelope

```json
{
  "protocolVersion": 1,
  "type": "RoomStateUpdated",
  "eventId": "evt_01J9R2X1QK0S9TE4YZ6A9N18KF",
  "requestId": "req_01J9R2WY4R2Q0M4E8VJ7A3N6KD",
  "sentAt": "2026-04-15T12:00:01Z",
  "stateVersion": 7,
  "payload": {}
}
```

Field rules:
- `protocolVersion`: increment only for breaking wire changes.
- `type`: exact event name from this document.
- `requestId`: generated by the client for correlation. Server echoes it when the event is a direct response to a request.
- `eventId`: unique server event id for dedupe and logs.
- `sentAt`: ISO-8601 UTC timestamp.
- `stateVersion`: monotonic per-room integer. Present on every server event that includes room state.

## 3. Canonical Render State

The Android client should render primarily from `payload.state`.

Rules:
- Every successful server event except `Pong` includes a full `state` snapshot.
- `ErrorMessage` may include `state` when the server wants the client to recover by re-rendering the latest valid state.
- The client should replace its local screen model with the newest `state` when `stateVersion` increases.
- The client should not infer hidden game information. If a field is absent or redacted, the server does not want that player to see it yet.

### `state` shape

```json
{
  "stateVersion": 7,
  "serverTime": "2026-04-15T12:00:01Z",
  "room": {
    "roomCode": "Q7LM",
    "status": "LOBBY",
    "hostPlayerId": "player_01",
    "createdAt": "2026-04-15T11:59:55Z",
    "updatedAt": "2026-04-15T12:00:01Z",
    "settings": {
      "minPlayers": 2,
      "maxPlayers": 4,
      "targetScore": 5,
      "mode": "CLASSIC"
    },
    "players": [
      {
        "playerId": "player_01",
        "name": "Nova",
        "seatIndex": 0,
        "isHost": true,
        "connected": true,
        "ready": true,
        "score": 0,
        "isJudge": false,
        "hasSubmittedCard": false
      },
      {
        "playerId": "player_02",
        "name": "Rex",
        "seatIndex": 1,
        "isHost": false,
        "connected": true,
        "ready": false,
        "score": 0,
        "isJudge": false,
        "hasSubmittedCard": false
      }
    ]
  },
  "viewer": {
    "playerId": "player_01",
    "playerToken": "ptk_6d1d8e67f1d94f9fba4d2f7d",
    "permissions": {
      "canStartGame": true,
      "canToggleReady": true,
      "canRename": true,
      "canSubmitCard": false,
      "canJudgePick": false
    },
    "pendingAction": "WAITING_IN_LOBBY",
    "hand": [
      {
        "cardId": "white_102",
        "text": "an apology email drafted at 2:13 AM"
      },
      {
        "cardId": "white_231",
        "text": "a team-building exercise that became legally complicated"
      }
    ]
  },
  "round": null,
  "scoreboard": [
    {
      "playerId": "player_01",
      "name": "Nova",
      "score": 0,
      "rank": 1
    },
    {
      "playerId": "player_02",
      "name": "Rex",
      "score": 0,
      "rank": 1
    }
  ]
}
```

### `room.status`

- `LOBBY`
- `SUBMITTING`
- `JUDGING`
- `ROUND_RESULT`
- `GAME_OVER`

### `viewer.pendingAction`

- `WAITING_IN_LOBBY`
- `READY_UP`
- `WAITING_FOR_HOST`
- `SUBMIT_CARD`
- `WAITING_FOR_SUBMISSIONS`
- `JUDGE_PICK`
- `WAITING_FOR_JUDGE`
- `VIEW_ROUND_RESULT`
- `VIEW_FINAL_SCOREBOARD`

### `round` shape

When a game is active, `round` is present:

```json
{
  "matchId": "match_01",
  "roundId": "round_03",
  "roundNumber": 3,
  "phase": "SUBMITTING",
  "judgePlayerId": "player_02",
  "prompt": {
    "promptId": "black_089",
    "text": "Nothing says leadership like ____."
  },
  "submissionSummary": {
    "requiredCount": 2,
    "receivedCount": 1
  },
  "submissions": [
    {
      "submissionId": "sub_01",
      "cardId": "white_231",
      "text": "a team-building exercise that became legally complicated",
      "submittedByPlayerId": null,
      "isWinner": false
    }
  ],
  "winningSubmissionId": null,
  "winnerPlayerId": null
}
```

Visibility rules:
- In `SUBMITTING`, `submissions` only contains the local player's accepted submission, if any.
- In `JUDGING`, `submissions` contains all candidate answers, but `submittedByPlayerId` stays `null`.
- In `ROUND_RESULT`, `submissions` contains all answers and `submittedByPlayerId` is populated.
- In `GAME_OVER`, `round` may still hold the final round result or be `null`. The client should rely on `room.status`.

## 4. Client -> Server Events

All examples below show only the `payload` for brevity.

### `CreateRoom`

Creates a new room and host player.

```json
{
  "playerName": "Nova",
  "settings": {
    "targetScore": 5,
    "mode": "CLASSIC"
  },
  "clientInfo": {
    "platform": "android",
    "appVersion": "1.0.0"
  }
}
```

Validation:
- `playerName`: 2 to 20 visible characters
- `targetScore`: 3 to 10 for v1
- `mode`: `CLASSIC` or `DUEL`

### `JoinRoom`

Joins an existing room as a new player.

```json
{
  "roomCode": "Q7LM",
  "playerName": "Rex",
  "clientInfo": {
    "platform": "android",
    "appVersion": "1.0.0"
  }
}
```

### `ReconnectRoom`

Reclaims an existing player seat after disconnect or app restart.

```json
{
  "roomCode": "Q7LM",
  "playerToken": "ptk_6d1d8e67f1d94f9fba4d2f7d",
  "lastKnownStateVersion": 11
}
```

### `LeaveRoom`

Leaves the room voluntarily.

```json
{
  "roomCode": "Q7LM",
  "playerToken": "ptk_6d1d8e67f1d94f9fba4d2f7d"
}
```

### `ToggleReady`

Sets the desired ready state. The event name stays `ToggleReady` to match UI intent, but the payload uses an explicit target value so retries are idempotent.

```json
{
  "roomCode": "Q7LM",
  "playerToken": "ptk_6d1d8e67f1d94f9fba4d2f7d",
  "ready": true
}
```

### `UpdatePlayerName`

Updates the local player's display name.

```json
{
  "roomCode": "Q7LM",
  "playerToken": "ptk_6d1d8e67f1d94f9fba4d2f7d",
  "playerName": "Nova Prime"
}
```

### `StartGame`

Host-only action that starts the match.

```json
{
  "roomCode": "Q7LM",
  "playerToken": "ptk_6d1d8e67f1d94f9fba4d2f7d"
}
```

Validation:
- only host may start
- player count must be between 2 and 4
- every connected player must be ready

### `SubmitCard`

Submits one white card for the active prompt.

```json
{
  "roomCode": "Q7LM",
  "playerToken": "ptk_6d1d8e67f1d94f9fba4d2f7d",
  "roundId": "round_03",
  "cardId": "white_231"
}
```

Validation:
- only non-judge players may submit
- card must exist in the sender's current hand
- one submission per player for v1

### `JudgePick`

Judge selects the winning anonymous submission.

```json
{
  "roomCode": "Q7LM",
  "playerToken": "ptk_8b4a1d3b2fc7410ba7f58e0b",
  "roundId": "round_03",
  "submissionId": "sub_09"
}
```

Validation:
- only the current judge may send this
- phase must be `JUDGING`

### `RequestStateSync`

Asks the server to resend the latest authoritative snapshot.

```json
{
  "roomCode": "Q7LM",
  "playerToken": "ptk_6d1d8e67f1d94f9fba4d2f7d",
  "lastKnownStateVersion": 11,
  "reason": "APP_RESUMED"
}
```

Suggested `reason` values:
- `APP_RESUMED`
- `MISSED_EVENTS`
- `SCREEN_RECREATED`
- `MANUAL_RETRY`

### `Ping`

Connection keepalive and lightweight latency measurement.

```json
{
  "roomCode": "Q7LM",
  "playerToken": "ptk_6d1d8e67f1d94f9fba4d2f7d",
  "clientTimestamp": "2026-04-15T12:00:04Z",
  "lastKnownStateVersion": 11
}
```

## 5. Server -> Client Events

Every successful event below includes `payload.state`, unless explicitly noted otherwise.

### `RoomCreated`

Returned only to the room creator.

```json
{
  "roomCode": "Q7LM",
  "playerId": "player_01",
  "playerToken": "ptk_6d1d8e67f1d94f9fba4d2f7d",
  "state": {
    "stateVersion": 1
  }
}
```

### `RoomStateUpdated`

Generic room snapshot broadcast after lobby changes and non-phase-specific updates.

```json
{
  "reason": "PLAYER_JOINED",
  "state": {
    "stateVersion": 3
  }
}
```

Suggested `reason` values:
- `PLAYER_JOINED`
- `PLAYER_LEFT`
- `PLAYER_DISCONNECTED`
- `PLAYER_RECONNECTED`
- `READY_CHANGED`
- `NAME_CHANGED`
- `HOST_CHANGED`
- `STATE_SYNC`

### `GameStarted`

Broadcast when the server transitions from lobby to match.

```json
{
  "matchId": "match_01",
  "startedByPlayerId": "player_01",
  "state": {
    "stateVersion": 8
  }
}
```

### `PromptShown`

Broadcast when a new prompt is active and the round is accepting submissions.

```json
{
  "matchId": "match_01",
  "roundId": "round_01",
  "prompt": {
    "promptId": "black_089",
    "text": "Nothing says leadership like ____."
  },
  "submissionDeadlineSeconds": 45,
  "state": {
    "stateVersion": 9
  }
}
```

### `HandUpdated`

Sent to one player when their private hand changes. This usually happens at match start and after a round advances.

```json
{
  "playerId": "player_01",
  "hand": [
    {
      "cardId": "white_102",
      "text": "an apology email drafted at 2:13 AM"
    },
    {
      "cardId": "white_231",
      "text": "a team-building exercise that became legally complicated"
    },
    {
      "cardId": "white_332",
      "text": "a calendar invite labeled urgent but spiritually vague"
    }
  ],
  "drawCount": 3,
  "state": {
    "stateVersion": 9
  }
}
```

### `SubmissionAccepted`

Sent to the submitting player immediately after a valid card submission.

```json
{
  "roundId": "round_01",
  "submissionId": "sub_01",
  "submittedCardId": "white_231",
  "state": {
    "stateVersion": 10
  }
}
```

### `JudgePhaseStarted`

Broadcast when all required submissions are in and the judge can choose a winner.

```json
{
  "roundId": "round_01",
  "judgePlayerId": "player_02",
  "submissions": [
    {
      "submissionId": "sub_01",
      "cardId": "white_231",
      "text": "a team-building exercise that became legally complicated",
      "submittedByPlayerId": null,
      "isWinner": false
    },
    {
      "submissionId": "sub_02",
      "cardId": "white_112",
      "text": "an HR-approved panic spiral",
      "submittedByPlayerId": null,
      "isWinner": false
    }
  ],
  "state": {
    "stateVersion": 11
  }
}
```

### `WinnerChosen`

Broadcast when the judge selects the winning answer.

```json
{
  "roundId": "round_01",
  "winningSubmissionId": "sub_02",
  "winnerPlayerId": "player_03",
  "scoreAwarded": 1,
  "state": {
    "stateVersion": 12
  }
}
```

### `RoundAdvanced`

Broadcast when the next round has been initialized and the new judge/prompt are ready.

```json
{
  "previousRoundId": "round_01",
  "nextRoundId": "round_02",
  "nextJudgePlayerId": "player_03",
  "state": {
    "stateVersion": 13
  }
}
```

### `ReconnectedState`

Returned to a reconnecting player after token validation. This is the authoritative resume message.

```json
{
  "roomCode": "Q7LM",
  "playerId": "player_01",
  "playerToken": "ptk_6d1d8e67f1d94f9fba4d2f7d",
  "resumedFromStateVersion": 11,
  "state": {
    "stateVersion": 13
  }
}
```

### `GameOver`

Broadcast when a player reaches the target score.

```json
{
  "matchId": "match_01",
  "winnerPlayerId": "player_03",
  "finalStandings": [
    {
      "playerId": "player_03",
      "name": "Jade",
      "score": 5,
      "rank": 1
    },
    {
      "playerId": "player_01",
      "name": "Nova",
      "score": 3,
      "rank": 2
    }
  ],
  "state": {
    "stateVersion": 21
  }
}
```

### `ErrorMessage`

Sent when the client request is invalid, unauthorized, out of phase, or references stale state.

```json
{
  "code": "ROOM_NOT_FOUND",
  "message": "Room Q7LM does not exist.",
  "recoverable": true,
  "state": null
}
```

Suggested error codes:
- `ROOM_NOT_FOUND`
- `ROOM_FULL`
- `PLAYER_NAME_INVALID`
- `PLAYER_TOKEN_INVALID`
- `ALREADY_JOINED`
- `NOT_HOST`
- `NOT_READY_TO_START`
- `MATCH_ALREADY_STARTED`
- `ROUND_NOT_ACTIVE`
- `NOT_JUDGE`
- `CARD_NOT_IN_HAND`
- `ALREADY_SUBMITTED`
- `STATE_VERSION_TOO_OLD`
- `STATE_VERSION_AHEAD`
- `INTERNAL_ERROR`

### `Pong`

Lightweight ping response. This does not need a full room snapshot.

```json
{
  "clientTimestamp": "2026-04-15T12:00:04Z",
  "serverTimestamp": "2026-04-15T12:00:04Z",
  "stateVersion": 11
}
```

## 6. Rendering Contract

The Android client can drive every major screen from `payload.state`:

- Splash: no socket state yet
- Home: local-only screen
- Create/Join: local-only until `RoomCreated` or `RoomStateUpdated`
- Lobby: `room.status = LOBBY`
- Game submit screen: `room.status = SUBMITTING` and `viewer.pendingAction = SUBMIT_CARD`
- Judge screen: `room.status = JUDGING` and `viewer.pendingAction = JUDGE_PICK`
- Waiting screens: same room status with a different `viewer.pendingAction`
- Round result: `room.status = ROUND_RESULT`
- Final scoreboard: `room.status = GAME_OVER`

The client should avoid separate ad hoc booleans when these can be derived from `state`.

## 7. Server State Update Rules

Recommended behavior:
- Broadcast `RoomStateUpdated` for lobby changes.
- On major phase changes, send the named semantic event and include full `state`.
- Increment `stateVersion` on every room mutation.
- Tailor `state.viewer` and `state.round.submissions` to the receiving player.
- Send `HandUpdated` only to the affected player, but still include full `state`.
- If a client reconnects or detects a gap, answer with `ReconnectedState` or `RoomStateUpdated` from `RequestStateSync`.

## 8. Typical Event Flow

### Room setup

1. Client sends `CreateRoom`
2. Server replies `RoomCreated`
3. Other players send `JoinRoom`
4. Server broadcasts `RoomStateUpdated`
5. Players send `ToggleReady`
6. Server broadcasts `RoomStateUpdated`
7. Host sends `StartGame`
8. Server broadcasts `GameStarted`
9. Server broadcasts `PromptShown`
10. Server sends `HandUpdated` to each player

### Round resolution

1. Each non-judge player sends `SubmitCard`
2. Server replies `SubmissionAccepted` to each submitting player
3. Server broadcasts `RoomStateUpdated` as submission counters change, or skips straight to `JudgePhaseStarted`
4. Judge sends `JudgePick`
5. Server broadcasts `WinnerChosen`
6. Server broadcasts `RoundAdvanced` or `GameOver`

### Resume after disconnect

1. Client reconnects WebSocket
2. Client sends `ReconnectRoom`
3. Server validates `playerToken`
4. Server returns `ReconnectedState`

## 9. Implementation Notes

Suggested server-side modeling:
- keep a room aggregate as the source of truth
- generate a recipient-specific `state` snapshot per outbound event
- keep player tokens opaque and random
- treat `requestId` as client correlation only, not identity
- log `eventId`, `requestId`, `roomCode`, `playerId`, and `stateVersion` for debugging

Suggested Android modeling:
- one immutable `GameUiState` mapped from the server `state`
- one WebSocket event parser
- ViewModels react to envelopes and replace current state on newer `stateVersion`
- screens derive visibility from `room.status` and `viewer.pendingAction`
